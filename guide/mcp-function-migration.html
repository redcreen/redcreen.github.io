🔌 MCP 是什么，以及把现有 function tools 改造成 MCP server 的迁移思路

从协议角色、工作机制，到实际改造路径，一次性讲清楚

这篇页面按原始 Markdown 文稿展开，重点是把三个问题放在同一条线里解释清楚：MCP 到底是什么、它和普通 function calling 有什么区别、以及怎么把已有 function tools 逐步迁移成 MCP server。

1. MCP 是什么

MCP 全称是 Model Context Protocol。可以把它理解成：

给 LLM / Agent 接外部工具和数据源的统一接口标准。

以前如果要让模型访问数据库、Slack、GitHub、文件系统、内部业务服务，通常每一个系统都要单独写一套集成逻辑。

MCP 想解决的是：

  • 不同 AI 应用和工具服务之间有一个统一协议
  • 工具提供方实现一次协议后，可以被多个 AI 客户端复用
  • 模型访问外部世界时，不再依赖每家平台各自的私有接法

一句话总结：

MCP 不是模型本身，而是模型和外部工具 / 数据之间的标准化协议层。

2. MCP 的核心角色

MCP 里最常见的三个角色是：

  • Host
    • 最终承载 AI 的应用
    • 比如桌面客户端、IDE agent、聊天应用、某个多 agent 系统
  • MCP Client
    • Host 里的协议客户端
    • 负责和 MCP Server 通信
  • MCP Server
    • 你自己写的，或第三方提供的服务
    • 对外暴露工具、资源和提示模板

可以把它想成：

  • Host = AI 应用本体
  • Client = Host 里负责说 MCP 协议的那部分
  • Server = 真正提供能力的工具服务

3. MCP Server 可以暴露什么

MCP 的核心 primitives 主要有三类：

3.1 Tools

可执行动作，例如：

  • 查订单
  • 搜日志
  • 发消息
  • 改工单
  • 执行发布

3.2 Resources

可读上下文，例如：

  • 文档内容
  • 数据库 schema
  • 配置文件
  • API 说明
  • 某个只读数据对象

3.3 Prompts

可复用提示模板，例如：

  • 某种固定分析模板
  • 某种总结提示词
  • 某类工作流起始 prompt

所以不要把 MCP 简单理解成“只是工具调用协议”。它不只是 tools，也包括 resources 和 prompts。

4. MCP 的工作机制

最简单的理解是：

  1. AI 应用连接一个或多个 MCP Server
  2. Client 和 Server 完成初始化与能力协商
  3. Client 发现 Server 提供了哪些 tools / resources / prompts
  4. 用户发起任务后，模型决定要不要使用这些能力
  5. 如果需要：
    • 读资源：例如 resources/list、resources/read
    • 调工具：例如 tools/list、tools/call
    • 取提示模板：例如 prompts/list、prompts/get
  6. Server 返回结果
  7. Host 把结果再交给模型继续推理
  8. 模型输出最终结果

本质上就是：

模型不直接理解你的数据库和系统，
而是通过 MCP 去“发现能力 -> 调用能力 -> 拿回结果 -> 再继续推理”。

5. MCP 的传输方式

MCP 协议层基于 JSON-RPC 2.0。

常见传输层通常有两类：

5.1 stdio

适合：

  • 本地进程
  • 本机开发
  • IDE / CLI 集成

优点：

  • 简单
  • 调试方便
  • 不用先解决远程认证和部署问题

5.2 Streamable HTTP

适合：

  • 远程 MCP Server
  • 多客户端接入
  • 需要认证、网关和服务化部署

所以实际项目里很常见的路径是：

  • 先本地 stdio
  • 稳定后再上远程 HTTP

6. MCP 和普通 function calling 的区别

这个区别是最容易混淆的。

6.1 function calling 是什么

function calling 更像：

在一次模型请求里，告诉模型有哪些函数可以调。

典型流程：

  1. 你的应用把 tool / function 定义发给模型
  2. 模型返回：要调用哪个函数、参数是什么
  3. 你的应用执行函数
  4. 你的应用把结果再发回模型
  5. 模型继续输出

特点：

  • 工具定义往往是“这次请求里临时带给模型”
  • 执行位置通常在你自己的应用里
  • 更适合单应用内集成

6.2 MCP 是什么

MCP 更像：

AI 系统连接外部工具世界的一层标准协议。

特点：

  • 有 client / server 模型
  • 有能力发现
  • 有标准 transport
  • 不只是 tool，还包括 resource 和 prompt
  • 更适合做长期可复用的工具层

6.3 一句话区分

  • function calling：解决“这次调用里模型怎么调函数”
  • MCP：解决“AI 系统怎么标准化接外部世界”

7. 一个最小的 MCP Server 长什么样

下面是一个最小 Python 示例，使用 MCP Python SDK 的 FastMCP：

Python / FastMCP
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Demo Server")


@mcp.tool()
def get_weather(city: str) -> str:
    """Get weather for a city."""
    return f"{city}: sunny"


@mcp.resource("config://app")
def app_config() -> str:
    return '{"env":"dev","region":"cn"}'


@mcp.prompt()
def summarize_prompt(topic: str) -> str:
    return f"Please summarize the topic: {topic}"


if __name__ == "__main__":
    mcp.run()

这个示例做了三件事：

  • 用 @mcp.tool() 注册一个可执行工具
  • 用 @mcp.resource() 注册一个可读资源
  • 用 @mcp.prompt() 注册一个可复用提示模板

然后支持 MCP 的客户端接上来后，就能发现并使用这些能力。

8. 把现有 function tools 改造成 MCP server 的总体思路

最实用的结论是：

不要一开始就重写工具，
而是先保留原有业务函数，在外面包一层 MCP adapter。

这通常是迁移成本最低、风险最小的方案。

9. 先判断你现在的 function tools 属于哪一类

迁移前先分类，通常会分成三种：

9.1 纯函数型

例如：

  • get_order(order_id)
  • search_logs(query)

特点：

  • 输入明确
  • 输出明确
  • 不太依赖当前会话环境

这类最容易迁移。

9.2 带上下文适配型

例如：

  • 工具内部依赖当前用户
  • 工具内部依赖当前租户
  • 工具内部直接读全局 session

这类迁移时要特别注意权限边界。

9.3 面向单次 prompt 设计型

例如：

  • 参数很随意
  • 返回结构不稳定
  • 只为了某个特定 prompt 临时设计

这类最需要先收口接口。

10. 迁移前先做“盘点和拆分”

不是所有现有 function tool 都应该直接变成 MCP tool。

先把现有能力分成三类：

  • 保留为 tool
  • 更适合改成 resource
  • 应该继续留在应用层，不直接暴露给模型

判断规则：

  • 有副作用、能执行动作：tool
  • 只读上下文：resource
  • 纯提示模板：prompt

这是 MCP 和普通 function calling 很不一样的一点。

在 function calling 里，人们容易把所有东西都塞成 tool；在 MCP 里，更推荐按能力边界拆清楚。

11. 最推荐的迁移方式：先加适配层，不改原业务逻辑

假设你原来有这样的函数：

Python
def get_order(order_id: str) -> dict:
    ...

不要急着改它。

先在 MCP server 里包一层：

Python
@mcp.tool()
def get_order_tool(order_id: str) -> dict:
    return get_order(order_id)

也就是：

  • 原函数继续做业务实现
  • MCP adapter 层负责协议暴露

这样做的好处：

  • 原调用方不受影响
  • 迁移可以逐步进行
  • 出问题容易回退
  • 后面也能同时支持 function calling 和 MCP

12. 迁移时最应该统一的是输入输出 schema

很多旧的 function tools 会有这些问题：

  • 参数名不统一
  • 返回结构不稳定
  • 错误格式混乱
  • 一个工具的粒度太大

迁到 MCP 前，最好先统一：

  • 输入参数清晰
  • 输出结构稳定
  • 错误语义明确

例如，把这种函数：

Python
def search_logs(q, t=None, x=False):
    ...

收口成这种：

Python
def search_logs(query: str, time_range: str | None = None) -> dict:
    return {
        "results": [...],
        "count": 12,
    }

因为 MCP 更适合暴露“长期可复用的能力”，而不是只给某个 prompt 拼起来的一次性接口。

13. 推荐的代码分层

迁移时很推荐把代码整理成三层：

13.1 Domain layer

原始业务逻辑层：

Python
def get_order(order_id: str) -> dict: ...
def search_logs(query: str) -> dict: ...

13.2 Service layer

统一做：

  • 输入校验
  • 权限判断
  • 错误处理
  • 输出格式整理

例如：

Python
def get_order_service(order_id: str, actor: Actor) -> dict: ...

13.3 MCP adapter layer

只负责把能力暴露给 MCP：

Python
@mcp.tool()
def get_order_tool(order_id: str) -> dict:
    actor = resolve_actor()
    return get_order_service(order_id, actor)

这种分层的好处是：

  • 原来的业务逻辑还可以被别的代码复用
  • 以后也能同时支持 MCP、HTTP API、function calling
  • 权限和协议不会混在一起

14. 推荐的迁移顺序

不要一口气迁全部工具。

更稳的顺序通常是：

  1. 先盘点现有 tools，分成 tool / resource / prompt
  2. 先挑 2 到 3 个只读查询工具试点
  3. 统一输入输出 schema
  4. 本地先跑 stdio MCP server
  5. 在 agent 侧同时保留旧 function calling 和新 MCP，一段时间并行
  6. 稳定后再迁远程部署、认证、审计
  7. 最后再迁写操作类工具

原因很简单：

  • 查询类风险低
  • 写操作类会涉及权限、审计、幂等、确认

15. 哪些 function tools 不适合直接搬

以下几类要特别小心：

15.1 强依赖当前 prompt 的工具

例如：

  • 参数是 prompt 里隐式约定的
  • 返回值只适配某一个 agent 的说话方式

这类要先去掉 prompt 耦合。

15.2 隐式拿全局状态的工具

例如：

  • 默认读当前 session
  • 默认拿当前登录用户
  • 默认走某个全局配置

这类如果不收口，很容易在 MCP 场景下变成权限漏洞。

15.3 粗粒度过大的工具

例如：

  • do_everything(query)
  • 一个工具里同时做搜索、判断、执行、总结

这类更适合拆成几个清晰能力：

  • search_x
  • fetch_x
  • update_x

16. 如果目标是接 ChatGPT / deep research，要额外注意

如果你后面要把 MCP server 接到 OpenAI 生态里，尤其是 ChatGPT apps 或 deep research 相关场景，通常要特别注意：

  • 搜索型能力可以优先收敛成 search
  • 获取详情型能力可以优先收敛成 fetch

这样兼容性往往更好。

也就是说，如果你原来的 tool 叫：

  • search_docs
  • get_doc_by_id

那迁移到 MCP 时，可以优先考虑统一成：

  • search
  • fetch

17. 一条最实用的设计原则

可以把整个迁移过程压缩成一句话：

保留原业务函数，
新增一层 MCP adapter，
先迁只读工具，
先统一 schema，
最后再迁高风险写操作。

这个思路通常比“直接重写一套 MCP server 版本的业务逻辑”更稳。

18. 总结

18.1 关于 MCP

  • MCP 是标准化协议层
  • 它让 AI 应用以统一方式访问工具、资源和提示模板
  • 它不只是 tool 调用，还包括 resources 和 prompts

18.2 关于 function calling

  • function calling 更像单次请求里的函数调用能力
  • 更适合单应用内快速集成

18.3 关于迁移

  • 迁移重点不是“换个注册方式”
  • 迁移重点是：
    • 能力边界清晰化
    • 协议标准化
    • 输入输出稳定化
    • 工具和 prompt 解耦

18.4 最终一句话

小而快的集成，用 function calling；
长期可复用的工具层，用 MCP；
真正落地迁移时，先包 adapter，不要先重写业务。

19. tool 和 MCP 的区别

最简单说：

tool 是“能力本身”；
MCP 是“暴露和访问这些能力的协议/标准”。

19.1 tool 是什么

一个 tool 就是一项可调用能力，例如：

  • get_order
  • search_logs
  • send_slack_message

它关注的是：

  • 做什么
  • 输入是什么
  • 输出是什么

19.2 MCP 是什么

MCP 关注的是：

  • 这些能力怎么被标准化暴露出来
  • AI 应用怎么发现它们
  • 怎么调用它们
  • 怎么返回结果

所以：

  • tool 是业务能力
  • MCP 是工具接入协议

19.3 一个很实用的类比

  • tool 像一个电器
  • MCP 像统一插座标准

或者：

  • tool 像一个 API endpoint
  • MCP 像 API 协议和接口规范

一句话总结：

tool 回答“能做什么”；
MCP 回答“这些能力怎么以统一方式被发现、调用和复用”。

20. function tool、MCP tool 和 hosted tool 的关系

20.1 function tool

最常见的形态是：

  • 你的应用把 tool 定义直接塞给 LLM
  • 模型返回要调用哪个 tool、参数是什么
  • 你的应用自己执行
  • 再把结果回给模型

它更像：

应用内直接函数分发。

20.2 MCP tool

最常见的形态是：

  • Host / agent 先连接 MCP server
  • 通过协议发现有哪些 tools
  • 再把这些能力提供给模型
  • 模型决定调用后，由 Host 路由到 MCP server
  • MCP server 返回结果
  • Host 再把结果交回模型

它更像：

通过标准协议调一个外部能力服务。

20.3 hosted tool

这类是平台自己提供并执行的工具，例如：

  • web_search
  • file_search
  • code_interpreter
  • computer_use

特点：

  • 工具由平台官方提供
  • 执行位置通常在平台侧
  • 不是你自己托管的业务代码

20.4 一句话区别

  • function tool：你把 tool 直接塞给模型
  • MCP tool：你把 tool 挂到 MCP server 上，再由 Host 提供给模型
  • hosted tool：平台自己提供并执行的工具

最关键的一句是：

function tool 和 MCP tool 的核心区别，不在于是不是 tool，而在于 tool 是怎么暴露给模型的。

21. LLM 如何发现 MCP tool，并决定怎么调用

这里一个很关键的点是：

严格来说，不是 LLM 自己主动去“网上发现” MCP tool，
而是 Host / agent 先发现，再提供给 LLM。

最常见流程是：

  1. Host / agent 先连接 MCP server
  2. 通过 MCP 协议拿到：
    • 有哪些 tools
    • 它们的名字
    • 描述
    • 参数 schema
  3. Host / agent 再把这些工具能力提供给 LLM
  4. LLM 基于这些描述，决定要不要调用、调用哪个、参数怎么填

所以从工程视角，可以先这样理解：

function tool 是 agent 直接塞给 LLM；
MCP tool 是 agent 先从 MCP server 发现并注册，再塞给 LLM。

这个理解在大部分设计讨论里已经足够用了。

21.1 LLM 真正根据什么决定调用

不管是 function tool 还是 MCP tool，模型主要看这些信息：

  • tool 名称
  • tool 描述
  • 参数 schema
  • 当前用户任务
  • 之前 tool 调用返回的结果

所以模型“决定怎么调用”的依据，本质上不是 MCP 本身，而是：

Host 暴露给它的工具元信息是否清晰、稳定、可理解。

22. LLM 实际调用 function tool 和 MCP tool 时，有什么区别

如果只从模型的推理动作看，差异其实没有那么大。

不管是哪一类 tool，模型都在做类似的事：

  1. 判断要不要用 tool
  2. 选哪个 tool
  3. 组织参数
  4. 发起调用
  5. 读取结果后继续推理

所以对模型来说，两者都很像是在“使用一个外部能力”。

22.1 真正不同的是系统侧链路

function tool 的典型链路：

Sequence
LLM
 -> 选择 tool
 -> 返回 tool name + args
 -> 应用本地执行
 -> 结果回给 LLM

MCP tool 的典型链路：

Sequence
LLM
 -> 选择 MCP tool
 -> Host / MCP client 路由
 -> MCP server 执行
 -> 结果回给 Host
 -> 再回给 LLM

所以区别不是“模型会不会调”，而是：

  • function tool：更像应用内函数调用
  • MCP tool：更像标准协议路由到工具服务

22.2 从模型效果上最大的区别

如果硬要说模型最有感的差异，通常是：

  • MCP tool 往往更稳定、更长期
  • function tool 往往更临时、更依赖当前应用上下文

也就是说，MCP tool 通常更容易形成稳定命名、稳定参数和稳定返回结构，模型更容易“学会怎么用它”。

23. 既然最后很多时候都是塞给 LLM，MCP 真正多出来的价值是什么

这个问题其实是 MCP 的关键。

一句话：

MCP 的价值不在“模型会不会调 tool”，
而在“工具层能不能标准化、复用、解耦、独立演进”。

23.1 让工具从“应用内私有函数”变成“可复用能力服务”

function tool 更像：

  • 某个应用里的内部函数
  • 常常跟当前 prompt 或当前 agent 绑得很死

MCP 更像：

  • 独立 server
  • 标准协议暴露
  • 多个 agent / 客户端可复用

23.2 让“接工具”这件事标准化

没有 MCP 时，经常会出现：

  • A 应用写一套 GitHub 集成
  • B 应用再写一套 GitHub 集成
  • C agent 平台又写一套 GitHub 集成

MCP 的价值就是减少这种重复造轮子。

23.3 更适合多工具、多系统环境

如果你只有两三个工具，function tool 往往够用。

但如果你有很多系统：

  • Slack
  • GitHub
  • PostgreSQL
  • 内部 CRM
  • 审批系统
  • 文档库

那 MCP 更适合作为统一接入层。

23.4 更适合独立演进和团队协作

MCP 更适合这种边界：

  • 平台组维护 Host / agent
  • 业务组维护订单工具
  • 数据组维护检索工具
  • 运维组维护权限和部署

也就是说，MCP 更像组织级的接口规范。

23.5 不只是 tool

MCP 原生就有：

  • tools
  • resources
  • prompts

所以它比“只会调函数”的方案更完整，更像一个 AI 集成协议，而不是单一工具调用机制。

23.6 什么情况下 MCP 不值得上

如果你只是：

  • 一个单体应用
  • 几个简单函数
  • 没有多客户端复用需求
  • 没有长期平台化需求

那 MCP 很可能是过度设计。这时 function tool 往往更快、更简单。

24. 本机调用时，MCP 的性能损失大概有多少

一个很实用的结论是：

本机调用时，MCP 确实会有额外开销，
但通常不是主要瓶颈。

尤其当 tool 本身就要：

  • 查数据库
  • 读文件
  • 搜日志
  • 发网络请求

这时 MCP 这层协议和进程开销，通常会被真实业务耗时淹没。

24.1 本机 MCP 多出来的开销是什么

如果是本机 stdio MCP，额外开销通常来自：

  1. 进程边界
  2. JSON-RPC 序列化 / 反序列化
  3. 路由与调度

所以和本地直接 function call 相比，肯定会慢一点。

24.2 一般会慢多少

经验上可以先这样理解：

  • 本地直接函数调用：最低成本
  • 本机 stdio MCP：多一层轻量 IPC + JSON 编解码
  • 额外成本通常是毫秒级，甚至更低毫秒级

也就是说：

  • 如果 tool 本身耗时小于 1ms，这层开销会比较显眼
  • 如果 tool 本身要 50ms、200ms、1s，这层开销通常不是关键问题

24.3 什么场景下会明显感觉到慢

  • 超高频、小颗粒调用
  • 工具拆得太碎
  • 每次都重新启动 MCP server

真正拖垮性能的，往往不是 MCP 本身，而是：

  • 工具设计过碎
  • 调用次数过多
  • LLM 往返轮次太多

24.4 一个很实用的性能建议

  • 高频、超轻量逻辑：留在应用内，直接函数调用
  • 中低频、可复用、跨系统能力：做成 MCP
  • 不要把一个业务动作拆成过多微小 tool
  • 让 MCP server 常驻，不要反复拉起

一句话总结：

本机 MCP 相比直接 function call 确实有损耗，但通常是小开销；
只要 tool 不是超轻量高频调用，实际瓶颈一般不在 MCP。

25. 什么样的能力适合留在本地函数，什么样的能力适合抽成 MCP

25.1 更适合留在本地函数的能力

这类通常不值得抽成 MCP：

A. 超轻量工具

例如：

  • 字符串处理
  • JSON 格式整理
  • 简单字段映射
  • 很小的内存查询

特点：

  • 计算很轻
  • 调用频率高
  • 如果走 MCP，固定开销占比会偏大

B. 强依赖当前应用上下文

例如：

  • 直接依赖当前 session
  • 直接依赖当前页面状态
  • 直接依赖当前 agent 内部缓存

特点：

  • 很难做成通用服务
  • 抽出去后上下文传递反而更麻烦

C. 只服务某一个 agent / 某一个 prompt 的临时逻辑

例如：

  • 某个专用格式转换
  • 某个 agent 专属后处理
  • 某个实验性分支判断

特点：

  • 复用价值低
  • 生命周期短
  • 留在本地更简单

D. 高频基础组件

例如：

  • 参数校验
  • 小型规则判断
  • 局部数据拼装

特点：

  • 这些更像内部函数，不像独立工具服务

25.2 更适合抽成 MCP 的能力

A. 跨系统能力

例如：

  • 查 GitHub
  • 查 Slack
  • 查 PostgreSQL
  • 调内部业务系统 API

特点：

  • 本来就是相对独立的外部能力
  • 很适合标准化暴露

B. 多个 agent / 多个应用都会用的能力

例如：

  • 用户查询
  • 订单查询
  • 日志检索
  • 文档搜索

特点：

  • 复用价值高
  • 做成 MCP 后更容易共享

C. 需要独立演进的能力

例如：

  • 搜索服务
  • 检索服务
  • 审批服务
  • 内部知识库服务

特点：

  • 工具层和 agent 层解耦更有价值
  • 便于团队分工维护

D. 边界清晰的读写能力

例如：

  • search
  • fetch
  • create_ticket
  • update_status

特点：

  • 输入输出稳定
  • 容易定义 schema
  • 适合长期暴露给模型

E. 未来想平台化的能力

例如：

  • 公司统一知识检索
  • 公司统一工单能力
  • 公司统一业务查询接口

特点：

  • 现在可能只有一个 agent 在用
  • 但未来明显会变成公共能力

25.3 一个最实用的判断公式

你可以直接问四个问题：

  1. 这个能力是不是会被多个 agent / 应用复用？
  2. 它是不是一个相对独立的系统能力？
  3. 它的输入输出是不是稳定、清晰？
  4. 它值不值得独立维护和演进？

如果其中三项以上答案是“是”，那通常很适合抽成 MCP。

如果多数答案是“否”，那通常更适合留在本地函数。

25.4 最后一句经验

把“业务能力”抽成 MCP，
把“胶水逻辑”和“超轻量逻辑”留在本地函数。

例如：

  • 查询订单、查日志、搜文档：适合 MCP
  • 拼 prompt、改字段名、做小判断：适合本地函数
