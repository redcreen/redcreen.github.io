🔌 MCP 是什么，以及把现有 function tools 改造成 MCP server 的迁移思路

从协议角色、工作机制，到实际改造路径，一次性讲清楚

这篇页面按原始 Markdown 文稿展开，重点是把三个问题放在同一条线里解释清楚：MCP 到底是什么、它和普通 function calling 有什么区别、以及怎么把已有 function tools 逐步迁移成 MCP server。

1. MCP 是什么

MCP 全称是 Model Context Protocol。可以把它理解成：

给 LLM / Agent 接外部工具和数据源的统一接口标准。

以前如果要让模型访问数据库、Slack、GitHub、文件系统、内部业务服务，通常每一个系统都要单独写一套集成逻辑。

MCP 想解决的是：

  • 不同 AI 应用和工具服务之间有一个统一协议
  • 工具提供方实现一次协议后，可以被多个 AI 客户端复用
  • 模型访问外部世界时，不再依赖每家平台各自的私有接法

一句话总结：

MCP 不是模型本身，而是模型和外部工具 / 数据之间的标准化协议层。

2. MCP 的核心角色

MCP 里最常见的三个角色是：

  • Host
    • 最终承载 AI 的应用
    • 比如桌面客户端、IDE agent、聊天应用、某个多 agent 系统
  • MCP Client
    • Host 里的协议客户端
    • 负责和 MCP Server 通信
  • MCP Server
    • 你自己写的，或第三方提供的服务
    • 对外暴露工具、资源和提示模板

可以把它想成：

  • Host = AI 应用本体
  • Client = Host 里负责说 MCP 协议的那部分
  • Server = 真正提供能力的工具服务

3. MCP Server 可以暴露什么

MCP 的核心 primitives 主要有三类：

3.1 Tools

可执行动作，例如：

  • 查订单
  • 搜日志
  • 发消息
  • 改工单
  • 执行发布

3.2 Resources

可读上下文，例如：

  • 文档内容
  • 数据库 schema
  • 配置文件
  • API 说明
  • 某个只读数据对象

3.3 Prompts

可复用提示模板，例如：

  • 某种固定分析模板
  • 某种总结提示词
  • 某类工作流起始 prompt

所以不要把 MCP 简单理解成“只是工具调用协议”。它不只是 tools，也包括 resources 和 prompts。

4. MCP 的工作机制

最简单的理解是：

  1. AI 应用连接一个或多个 MCP Server
  2. Client 和 Server 完成初始化与能力协商
  3. Client 发现 Server 提供了哪些 tools / resources / prompts
  4. 用户发起任务后，模型决定要不要使用这些能力
  5. 如果需要：
    • 读资源：例如 resources/list、resources/read
    • 调工具：例如 tools/list、tools/call
    • 取提示模板：例如 prompts/list、prompts/get
  6. Server 返回结果
  7. Host 把结果再交给模型继续推理
  8. 模型输出最终结果

本质上就是：

模型不直接理解你的数据库和系统，
而是通过 MCP 去“发现能力 -> 调用能力 -> 拿回结果 -> 再继续推理”。

5. MCP 的传输方式

MCP 协议层基于 JSON-RPC 2.0。

常见传输层通常有两类：

5.1 stdio

适合：

  • 本地进程
  • 本机开发
  • IDE / CLI 集成

优点：

  • 简单
  • 调试方便
  • 不用先解决远程认证和部署问题

5.2 Streamable HTTP

适合：

  • 远程 MCP Server
  • 多客户端接入
  • 需要认证、网关和服务化部署

所以实际项目里很常见的路径是：

  • 先本地 stdio
  • 稳定后再上远程 HTTP

6. MCP 和普通 function calling 的区别

这个区别是最容易混淆的。

6.1 function calling 是什么

function calling 更像：

在一次模型请求里，告诉模型有哪些函数可以调。

典型流程：

  1. 你的应用把 tool / function 定义发给模型
  2. 模型返回：要调用哪个函数、参数是什么
  3. 你的应用执行函数
  4. 你的应用把结果再发回模型
  5. 模型继续输出

特点：

  • 工具定义往往是“这次请求里临时带给模型”
  • 执行位置通常在你自己的应用里
  • 更适合单应用内集成

6.2 MCP 是什么

MCP 更像：

AI 系统连接外部工具世界的一层标准协议。

特点：

  • 有 client / server 模型
  • 有能力发现
  • 有标准 transport
  • 不只是 tool，还包括 resource 和 prompt
  • 更适合做长期可复用的工具层

6.3 一句话区分

  • function calling：解决“这次调用里模型怎么调函数”
  • MCP：解决“AI 系统怎么标准化接外部世界”

7. 一个最小的 MCP Server 长什么样

下面是一个最小 Python 示例，使用 MCP Python SDK 的 FastMCP：

Python / FastMCP
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("Demo Server")


@mcp.tool()
def get_weather(city: str) -> str:
    """Get weather for a city."""
    return f"{city}: sunny"


@mcp.resource("config://app")
def app_config() -> str:
    return '{"env":"dev","region":"cn"}'


@mcp.prompt()
def summarize_prompt(topic: str) -> str:
    return f"Please summarize the topic: {topic}"


if __name__ == "__main__":
    mcp.run()

这个示例做了三件事：

  • 用 @mcp.tool() 注册一个可执行工具
  • 用 @mcp.resource() 注册一个可读资源
  • 用 @mcp.prompt() 注册一个可复用提示模板

然后支持 MCP 的客户端接上来后，就能发现并使用这些能力。

8. 把现有 function tools 改造成 MCP server 的总体思路

最实用的结论是：

不要一开始就重写工具，
而是先保留原有业务函数，在外面包一层 MCP adapter。

这通常是迁移成本最低、风险最小的方案。

9. 先判断你现在的 function tools 属于哪一类

迁移前先分类，通常会分成三种：

9.1 纯函数型

例如：

  • get_order(order_id)
  • search_logs(query)

特点：

  • 输入明确
  • 输出明确
  • 不太依赖当前会话环境

这类最容易迁移。

9.2 带上下文适配型

例如：

  • 工具内部依赖当前用户
  • 工具内部依赖当前租户
  • 工具内部直接读全局 session

这类迁移时要特别注意权限边界。

9.3 面向单次 prompt 设计型

例如：

  • 参数很随意
  • 返回结构不稳定
  • 只为了某个特定 prompt 临时设计

这类最需要先收口接口。

10. 迁移前先做“盘点和拆分”

不是所有现有 function tool 都应该直接变成 MCP tool。

先把现有能力分成三类：

  • 保留为 tool
  • 更适合改成 resource
  • 应该继续留在应用层，不直接暴露给模型

判断规则：

  • 有副作用、能执行动作：tool
  • 只读上下文：resource
  • 纯提示模板：prompt

这是 MCP 和普通 function calling 很不一样的一点。

在 function calling 里，人们容易把所有东西都塞成 tool；在 MCP 里，更推荐按能力边界拆清楚。

11. 最推荐的迁移方式：先加适配层，不改原业务逻辑

假设你原来有这样的函数：

Python
def get_order(order_id: str) -> dict:
    ...

不要急着改它。

先在 MCP server 里包一层：

Python
@mcp.tool()
def get_order_tool(order_id: str) -> dict:
    return get_order(order_id)

也就是：

  • 原函数继续做业务实现
  • MCP adapter 层负责协议暴露

这样做的好处：

  • 原调用方不受影响
  • 迁移可以逐步进行
  • 出问题容易回退
  • 后面也能同时支持 function calling 和 MCP

12. 迁移时最应该统一的是输入输出 schema

很多旧的 function tools 会有这些问题：

  • 参数名不统一
  • 返回结构不稳定
  • 错误格式混乱
  • 一个工具的粒度太大

迁到 MCP 前，最好先统一：

  • 输入参数清晰
  • 输出结构稳定
  • 错误语义明确

例如，把这种函数：

Python
def search_logs(q, t=None, x=False):
    ...

收口成这种：

Python
def search_logs(query: str, time_range: str | None = None) -> dict:
    return {
        "results": [...],
        "count": 12,
    }

因为 MCP 更适合暴露“长期可复用的能力”，而不是只给某个 prompt 拼起来的一次性接口。

13. 推荐的代码分层

迁移时很推荐把代码整理成三层：

13.1 Domain layer

原始业务逻辑层：

Python
def get_order(order_id: str) -> dict: ...
def search_logs(query: str) -> dict: ...

13.2 Service layer

统一做：

  • 输入校验
  • 权限判断
  • 错误处理
  • 输出格式整理

例如：

Python
def get_order_service(order_id: str, actor: Actor) -> dict: ...

13.3 MCP adapter layer

只负责把能力暴露给 MCP：

Python
@mcp.tool()
def get_order_tool(order_id: str) -> dict:
    actor = resolve_actor()
    return get_order_service(order_id, actor)

这种分层的好处是：

  • 原来的业务逻辑还可以被别的代码复用
  • 以后也能同时支持 MCP、HTTP API、function calling
  • 权限和协议不会混在一起

14. 推荐的迁移顺序

不要一口气迁全部工具。

更稳的顺序通常是：

  1. 先盘点现有 tools，分成 tool / resource / prompt
  2. 先挑 2 到 3 个只读查询工具试点
  3. 统一输入输出 schema
  4. 本地先跑 stdio MCP server
  5. 在 agent 侧同时保留旧 function calling 和新 MCP，一段时间并行
  6. 稳定后再迁远程部署、认证、审计
  7. 最后再迁写操作类工具

原因很简单：

  • 查询类风险低
  • 写操作类会涉及权限、审计、幂等、确认

15. 哪些 function tools 不适合直接搬

以下几类要特别小心：

15.1 强依赖当前 prompt 的工具

例如：

  • 参数是 prompt 里隐式约定的
  • 返回值只适配某一个 agent 的说话方式

这类要先去掉 prompt 耦合。

15.2 隐式拿全局状态的工具

例如：

  • 默认读当前 session
  • 默认拿当前登录用户
  • 默认走某个全局配置

这类如果不收口，很容易在 MCP 场景下变成权限漏洞。

15.3 粗粒度过大的工具

例如：

  • do_everything(query)
  • 一个工具里同时做搜索、判断、执行、总结

这类更适合拆成几个清晰能力：

  • search_x
  • fetch_x
  • update_x

16. 如果目标是接 ChatGPT / deep research，要额外注意

如果你后面要把 MCP server 接到 OpenAI 生态里，尤其是 ChatGPT apps 或 deep research 相关场景，通常要特别注意：

  • 搜索型能力可以优先收敛成 search
  • 获取详情型能力可以优先收敛成 fetch

这样兼容性往往更好。

也就是说，如果你原来的 tool 叫：

  • search_docs
  • get_doc_by_id

那迁移到 MCP 时，可以优先考虑统一成：

  • search
  • fetch

17. 一条最实用的设计原则

可以把整个迁移过程压缩成一句话：

保留原业务函数，
新增一层 MCP adapter，
先迁只读工具，
先统一 schema，
最后再迁高风险写操作。

这个思路通常比“直接重写一套 MCP server 版本的业务逻辑”更稳。

18. 总结

18.1 关于 MCP

  • MCP 是标准化协议层
  • 它让 AI 应用以统一方式访问工具、资源和提示模板
  • 它不只是 tool 调用，还包括 resources 和 prompts

18.2 关于 function calling

  • function calling 更像单次请求里的函数调用能力
  • 更适合单应用内快速集成

18.3 关于迁移

  • 迁移重点不是“换个注册方式”
  • 迁移重点是：
    • 能力边界清晰化
    • 协议标准化
    • 输入输出稳定化
    • 工具和 prompt 解耦

18.4 最终一句话

小而快的集成，用 function calling；
长期可复用的工具层，用 MCP；
真正落地迁移时，先包 adapter，不要先重写业务。
