🔀 多 Agent vs 多 Subagent

理解两种架构模式的本质差异，选择适合你场景的解决方案

在 OpenClaw 框架中，"多 Agent"和"多 Subagent"虽然听起来相似，但实际上是两种解决不同问题的架构模式。简单来说，"多 Agent"指的是多个各自独立的"常驻专家"，而"多 Subagent"指的是由一个"主管"临时派发的"项目临时工"。

📊 核心区别对比

Sub-agents

生命周期 永久运行

子 Agent 的生命周期与单个任务绑定，任务结束，它的会话也随之归档
配置方式 需要在 openclaw.json 配置文件中进行预先定义，例如指定 id、workspace、model 等 通常无需预先配置。由主 Agent 在运行时通过 sessions_spawn 命令动态创建，并可临时指定模型等参数
适用场景
  • 多身份/多角色：用一个 Agent 处理工作，另一个处理个人事务
  • 多租户隔离：为不同客户或群组分配完全隔离的 Agent
  • 长期值守任务：如一个 Agent 专门负责收集资讯，另一个负责定时发送报告
  • 复杂任务拆解：主 Agent 将一个大任务拆解为多个并行小任务，交给不同的子 Agent 同时执行
  • 成本优化：主 Agent 使用强大的高价模型做决策，子 Agent 使用廉价模型做具体的"体力活"
  • 上下文隔离：让子 Agent 在干净的上下文中专注执行，避免主会话的历史信息干扰
通信与协作 通过 sessions_send 工具进行"会话间通信"，就像两个独立的"人"在互相发消息 由主 Agent 通过 sessions_spawn 工具派发任务，子 Agent 执行完毕后将结果汇报给主 Agent

💡 如何选择？

根据你的具体需求，可以这样判断：

👤

个人用户，提高复杂任务效率

可以先用"1 个主 Agent + 多个 Subagent"的模式。把主 Agent 当作你的项目经理，由它去调度多个"临时工" Subagent 并行干活，这样架构简单、易于控制，也更省钱。

👥

多人共用，需要严格隔离

如果你的需求涉及多人共用、需要严格的身份或权限隔离，例如，你想让一个"客服 Bot"和一个"内部研发 Bot"拥有完全不同的权限和对话历史，那么就应该配置多个独立的持久化 Agent。

⚠️ 特别注意

在实际配置时，有几个容易踩的坑需要留意：

allowAgents 白名单位置

主 Agent 要调用哪些 Subagent，需要在配置文件的 agents.list 下具体的 Agent 条目中设置 subagents.allowAgents，不能放在 agents.defaults 全局默认值里，否则会导致网关启动失败。

功能冲突

sessions_spawn (子 Agent 功能) 和 agentToAgent.enabled: true (持久 Agent 间通信) 目前不能同时启用。如果同时开启，会导致子 Agent 创建后无法执行。你需要根据主要需求二选一。
