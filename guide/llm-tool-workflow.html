🧩 LLM、Tool 与状态机编排

把多 tool、状态机、平台托管和程序控制放到一张图里理解

这篇文章把一个常见混淆点讲清楚：“多 tool”说的是模型会不会调多个外部能力，“多次 LLM 调用”说的是流程是谁在驱动。 真实系统里，这两者经常同时存在。最实用的设计原则是：确定性的部分交给程序，语义判断和自然语言表达交给 LLM。

1. 先记住两个维度

维度 它描述什么 最简单的理解
多 Tool LLM 会不会调用多个外部能力 这是“能力使用方式”，比如查订单、查日志、再总结
多次 LLM 调用 整个流程是不是由程序分多轮驱动 这是“流程编排方式”，比如程序看状态后再决定什么时候调用下一轮 LLM
状态机 当前走到哪个阶段，当前允许做什么 状态机在程序里，不在 LLM 里。程序用它来限制当前可用 tool 和状态迁移

所以一个真实系统完全可能同时满足三件事：程序通过状态机多次调用 LLM，而每一轮 LLM 调用里又会继续使用多个 tool。

2. 为什么多 Tool 经常更快

大部分情况下，你可以先用这个心智模型理解：多 tool 往往更像“稳定前缀 + 增量追加”，平台更容易做状态复用和缓存，所以常常更快。

🔁

每次都重拼大 Prompt

如果每轮都重新带一遍 system prompt、原始需求、上一步结果和历史上下文，重复内容会很多。平台较难看出哪些内容其实没有变。

➕

在同一个任务里追加 Tool 结果

如果变化主要发生在尾部，前缀保持稳定，平台就更容易识别“前面相同、后面新增”，于是更容易做缓存、续跑和复用。

先别把原因理解得太底层

设计阶段不必纠结“是不是每一个 token 的底层计算都真的减少了”。更实用的结论是：多 tool 更容易形成稳定结构，所以在大部分平台实现里往往更快、更省重复开销。

3. 状态机和 LLM 怎么配合

状态机不是替代多 tool，而是在多 tool 外面再加一层“交通灯”。它不是让模型记住流程，而是由程序直接规定：你现在只能做这几步。

没有状态机

你一次性把全部 tool 都给模型，再在 prompt 里写“请按顺序做”。这种做法能跑，但本质还是软约束。

  • 📝规则在 prompt 里
  • 🤖模型负责遵守
  • ⚠️可能跳步或分支判断漂移

有状态机

程序记录当前状态，根据状态只开放有限 tool。LLM 不需要记完整流程，它只需要知道当前状态和当前允许做什么。

  • 🚦规则在程序里
  • 🔒系统强制边界
  • ✅更适合审批、审计、留痕
状态机 + LLM 的最小链路 
1. 程序读取当前 state
2. 程序把当前 state 和 allowed_tools 传给 LLM
3. LLM 在当前边界里选择一个 tool
4. 程序执行 tool
5. 程序根据 tool 结果切换 state
6. 程序再发起下一轮 LLM 调用

因此，只要是程序按状态机一轮一轮推进，本质上就是多次 LLM 调用。状态机和多 tool 不冲突，它们经常是叠在一起使用的。

4. 为什么不直接在程序里硬编码调用 Tool

如果调哪个 tool、按什么顺序调、tool 返回后怎么处理、分支条件怎么判断，这些事情都已经是确定的，那就没有必要额外绕一层 LLM。程序直接调 tool 往往更快、更稳、更便宜。

🧮

适合直接写死的部分

固定顺序、清晰规则、明确分支、结构化返回。比如 ETL、定时同步、固定审批节点的执行动作。

🗣️

适合交给 LLM 的部分

意图识别、tool 选择、长文本总结、日志归因、开放式语义判断、自然语言解释和追问。

最实用的原则：能写死，就别让 LLM 猜；需要理解和表达时，再让 LLM 参与。

5. 自定义 Tool 能不能放到平台端跑

这个问题最容易混淆。“模型能不能调用你的 tool”和“平台会不会替你执行你的业务代码”不是一回事。

类型 平台负责什么 真正执行位置
Function / Custom Tool 模型决定调哪个 tool、参数是什么 通常还是你自己的应用或服务
Remote MCP 模型通过标准接口使用你的远程工具 你自己部署的 MCP Server
Hosted Tools 平台直接提供能力，例如搜索或代码执行 OpenAI 平台侧

一个足够实用的判断

如果你问的是“模型能不能调用我定义的 tool”，答案通常是可以；如果你问的是“平台能不能替我托管任意自定义业务代码”，答案通常是否定的。自定义 tool 一般还是由你自己执行，或者由你自己部署成 MCP 服务。

6. 最后的设计建议

如果你只是想设计出一套大体合理的系统，不需要一开始就追求特别严格的理论模型。先按下面这套判断就够用了。

⚡

开放任务

优先考虑多 tool。让 LLM 在一个连续任务里读资料、查信息、调用外部能力，然后总结。

🚦

高风险流程

外层加状态机，程序卡住权限和步骤，再让 LLM 在当前状态内使用 tool。这样既保留灵活性，也不把边界全押在 prompt 上。

一句话总结：多 tool 负责效率和灵活性，状态机负责边界和可靠性，程序负责控制，LLM 负责理解与表达。
