🧩 LLM、Tool 与状态机编排

把多 tool、状态机、平台托管和程序控制放到一张图里理解

这篇页面基本按原始 Markdown 文稿展开，只做了网页排版层面的整理。目标不是做“再摘要一次”的二次创作，而是把关于多 tool、多次 LLM 调用、状态机和平台托管能力的整段讨论尽量完整地保留下来。

1. 先说结论

  • 多 tool 和 多次 LLM 调用 不是对立关系，而是两个不同维度。
  • 多 tool 说的是：LLM 会不会调用多个外部能力。
  • 多次 LLM 调用 说的是：整个流程是不是由程序分多轮驱动。
  • 一个系统完全可以同时满足这两点：
    • 程序通过状态机多次调用 LLM
    • 每次 LLM 调用里又可以调用多个 tool

最容易记的方式：

  • 多 tool = 能力使用方式
  • 多次 LLM 调用 = 流程编排方式

2. 多 Tool 和多次 LLM 调用的区别

2.1 多 Tool

典型形态：

  1. 用户给出一个任务
  2. LLM 判断先调用哪个 tool
  3. tool 返回结果
  4. LLM 再决定下一步是否继续调用别的 tool
  5. 最后输出结果

它更像一个 agent 在做事。

特点：

  • 路径更灵活
  • 对客户端编排要求更低
  • 适合“查 A -> 查 B -> 总结”这类开放型流程

2.2 多次 LLM 调用

典型形态：

  1. 程序先调用一次 LLM
  2. 根据结果调用 tool 或执行某个逻辑
  3. 程序再调用下一次 LLM
  4. 重复这个过程直到完成

它更像一个 workflow / controller 在驱动模型。

特点：

  • 控制力更强
  • 更适合固定流程
  • 更容易做审计、重试、人工介入、状态管理

3. 为什么很多场景下“多 Tool”看起来更省 token、更快

核心不是因为用了 tool 这两个字，而是因为：

多 tool 的工作流通常更像“稳定前缀 + 增量追加”，平台更容易做状态复用、缓存或续跑。

3.1 直觉理解

如果你每次都重新拼一个大 prompt：

  • system prompt 又带一遍
  • 用户问题又带一遍
  • 前面结果又带一遍
  • 历史上下文又带一遍

那重复内容会很多。

如果是同一个任务内不断追加 tool result：

  • 前缀通常比较稳定
  • 变化更多发生在尾部
  • 平台更容易识别“前面没变，后面只是多了一点新内容”

所以通常更容易：

  • 少重复传输
  • 少重复组装上下文
  • 更容易做缓存或状态复用
  • 在很多实现里表现得更快

3.2 但要注意

这个结论是“通常如此”，不是绝对规律。

因为还要看：

  • tool 返回是不是很大
  • 平台内部怎么计费
  • 平台是否真的做了缓存/摘要/复用
  • tool 本身是不是很慢

所以更准确的表达是：

大部分情况下，多 tool 更容易形成稳定上下文，因此平台更容易优化，所以经常更快。

4. “平台保存上下文”到底意味着什么

我们讨论里有一个重要澄清：

平台替你保存工作流现场，并不自动等于 LLM 实际处理的 token 计算量一定变少。

需要分开看三件事：

4.1 传输层

如果平台保存了状态，你后续可能只需要传：

  • previous_response_id
  • 新的 tool_output

这会让你客户端发出的内容明显变少。

4.2 计费层

很多时候会更省，但不一定完全等于“你没传就不算”。因为平台可能会在内部恢复上下文再喂给模型。

4.3 模型计算层

即使平台替你存了状态，模型在下一步推理时仍可能需要知道：

  • 前面发生了什么
  • 当前目标是什么
  • 上一个 tool 返回了什么

所以底层是否真的少算，取决于平台实现。

4.4 一个足够实用的理解

虽然不必追求特别严谨，但设计上可以先用这个心智模型：

多 tool 更容易形成稳定前缀和增量更新，所以更容易命中平台优化，通常更快。

这个理解对架构设计已经够用了。

5. 为什么“状态机 + 多 Tool”其实还是多次 LLM 调用

这是整段讨论里最关键的澄清点之一。

5.1 状态机是什么

状态机做的事很简单：

  1. 记录当前处于哪个阶段
  2. 决定当前允许哪些 tool
  3. 决定 tool 返回后如何切到下一个状态

例如：

  • INIT
  • WAITING_APPROVAL
  • APPROVED
  • REJECTED
  • DONE

5.2 它如何和 LLM 配合

通常是：

  1. 程序里维护状态机
  2. 程序读取当前状态
  3. 程序调用 LLM，并告诉它当前状态和当前允许的 tools
  4. LLM 调用某个 tool
  5. 程序执行 tool
  6. 程序根据 tool 结果更新状态
  7. 程序再次调用 LLM

所以只要是程序按状态机一轮一轮推进，本质上就是：

状态机编排 = 程序多次调用 LLM

这没有问题，也不和“多 tool”冲突。

5.3 为什么会容易混淆

因为“多 tool”描述的是 LLM 会不会调多个外部能力；而“状态机”描述的是流程由谁来控制。

所以真实系统里常见的是：

  • 外层：状态机 / workflow 控流程
  • 内层：LLM 在当前状态下使用多个 tool

6. 状态机卡流程，到底在“卡”什么

一句话：

不是让模型“记得按流程做”，而是程序直接限制它“现在只能这么做”。

6.1 不加状态机

假设你一次性把这些 tool 都给模型：

  • collect_info
  • human_approval
  • execute_action
  • reject_case

然后在 prompt 里写：

先收集信息，风险高就审批，通过后再执行。

这种做法的问题是：

  • 规则主要在 prompt 里
  • 模型是“被要求遵守”
  • 但系统没有从机制上强制

6.2 加状态机

程序按状态分配 tool：

  • INIT 只开放 collect_info
  • WAITING_APPROVAL 只开放 human_approval
  • APPROVED 只开放 execute_action
  • REJECTED 只开放 reject_case

这时模型在 WAITING_APPROVAL 阶段根本看不到 execute_action。

这就是“卡流程”。

所以：

  • 不加状态机：软约束
  • 加状态机：硬约束

7. 人工确认能不能也做成 Tool

可以，而且很常见。

例如可以做成：

  • request_human_approval
  • select_branch
  • confirm_risk_action

这样 LLM 也能在流程里调用“人工确认”。

但这里有一个关键区分：

7.1 人工确认作为 tool

  • 仍然是 LLM 主导流程
  • 模型决定什么时候问人、问什么、拿到结果后怎么继续

7.2 人工确认作为 workflow gate

  • 由程序控制“必须停在这里”
  • 没有审批结果就不允许进入下一步
  • 是否开放后续 tool 由程序决定

所以：

人工确认可以封成 tool，但如果流程要求很硬，外层仍然更适合由程序卡死。

8. 为什么不是直接在程序里硬编码调 Tool

这个问题其实是架构设计的核心。

如果以下事情都已经确定：

  • 调哪个 tool
  • 调用顺序
  • tool 返回后怎么处理
  • 分支条件也能用普通代码清楚表达

那通常没必要绕 LLM，程序直接调 tool 就好了。

这样一般会：

  • 更快
  • 更稳
  • 更便宜
  • 更容易测试

8.1 为什么还要引入 LLM

因为有些节点不适合硬编码，例如：

  1. 需要理解用户意图后再选 tool
  2. tool 返回的是长文本、日志、网页、文档，需要总结提炼
  3. 分支条件带语义判断，不是简单数值判断
  4. 需要自然语言解释、追问、总结、约束表达

所以一个很实用的原则是：

确定性的部分交给程序；
含糊的、语义性的、需要语言理解和生成的部分交给 LLM。

更白话一点：

能写死，就别让 LLM 猜。
需要“理解”和“表达”时，再用 LLM。

9. 什么时候选哪种方案

9.1 纯多 Tool / agent 风格

适合：

  • 开放式任务
  • 工具选择不完全固定
  • 目标是快速做事
  • 希望减少程序编排复杂度

例如：

  • 查资料
  • 读文件
  • 查日志
  • 最后总结

9.2 状态机 + 多 Tool

适合：

  • 有固定阶段
  • 流程不能跳步
  • 需要审批、审计、留痕
  • 某些动作存在权限边界或合规要求

例如：

  • 报销审批
  • SQL 执行审批
  • 发版审批流
  • 高风险操作执行

9.3 直接程序调用 Tool，不走 LLM

适合：

  • 流程完全确定
  • 分支规则简单明确
  • 不需要语义理解
  • 不需要自然语言生成

例如：

  • 定时同步数据
  • 按规则批处理
  • 固定 ETL
  • 明确的系统集成流程

10. 最推荐的设计方式：混合式

最稳的设计通常不是“全靠 prompt”，也不是“所有事情都写死”。

更常见、也更合理的做法是：

  1. 外层由程序维护状态机
  2. 每个状态只开放允许的 tools
  3. LLM 负责当前状态内的理解、判断、tool 选择和结果表达
  4. 状态切换由程序根据规则或外部事件触发

这可以概括成一句话：

程序控制“能不能做”，LLM 负责“怎么理解、怎么表达、在允许范围内怎么做”。

11. 一份最终可直接记住的结论

11.1 关于多 Tool 和多次 LLM 调用

  • 多 tool 不是 多次 LLM 调用 的反义词
  • 它们是两个不同维度
  • 真实系统里常常同时存在

11.2 关于速度

  • 大部分情况下，多 tool 工作流更容易形成稳定上下文
  • 平台更容易做状态复用和缓存
  • 所以常常更快

11.3 关于状态机

  • 状态机在程序里，不在 LLM 里
  • 程序按状态多次调用 LLM
  • 每次只给 LLM 当前状态允许的 tools
  • 状态变更通常由程序根据 tool 结果或外部事件触发

11.4 关于是否需要 LLM

  • 如果流程和分支都很确定，直接程序调用 tool
  • 如果需要语义判断、意图识别、文本总结、自然语言交互，再引入 LLM

11.5 最实用的一句话

能写死，就别让 LLM 猜；
需要理解和表达时，再让 LLM 参与；
高风险流程外层用状态机卡住，内层再给 LLM 多 tool 能力。

12. 自定义 Tool 能不能放到平台端跑

这个问题很容易和“平台支持 tool 调用”混在一起，实际上要分开看。

12.1 一般情况下，不能把任意自定义业务 Tool 直接丢给平台托管执行

更准确地说：

  • 你可以把自定义 tool 暴露给模型
  • 模型可以决定何时调用它
  • 但这个 tool 的真正执行位置，通常还是在你自己的服务、应用或 runtime 里

也就是说：

平台可以帮你做“tool decision”，
但不等于平台会替你执行任意自定义业务代码。

12.2 常见的三种情况

A. Function calling / custom tool

这是最常见的“自定义 tool”。

特点：

  • 你把 tool 定义告诉模型
  • 模型返回要调用哪个 tool、参数是什么
  • 真正执行还是在你的程序里

所以它本质上是：

模型负责决定调用，
你的程序负责执行。

B. Remote MCP

这也能让模型调用你的能力。

特点：

  • 你把能力做成一个 MCP server
  • 模型可以通过 MCP 使用这些工具
  • 但 MCP server 仍然是你自己部署和维护的服务

所以它不是“OpenAI 平台替你跑业务代码”，而是：

OpenAI 平台连接到你自己的工具服务。

C. 平台托管的 hosted tools

这类才更接近“平台端执行”。

例如官方提供的：

  • web_search
  • file_search
  • code_interpreter
  • image_generation
  • computer_use
  • shell

这些是平台自带能力，不是你把任意自定义函数上传上去运行。

12.3 一个很实用的判断方式

如果你问的是：

“模型能不能调用我定义的 tool？”

答案通常是：

可以。

如果你问的是：

“OpenAI 能不能替我托管执行任意自定义业务 tool 代码？”

答案通常是：

不行，通常还是你自己执行，或者你自己部署成 MCP 服务。

12.4 设计上的启发

所以实际架构里，一般是这样分层：

  • OpenAI 平台负责模型推理和工具调用决策
  • 你的应用负责业务 tool 执行
  • 如果要远程化、标准化，可以把工具封装成 MCP server
  • 如果需求刚好能被官方 hosted tools 覆盖，就直接用平台能力

一句话总结就是：

自定义 tool 可以接入平台让模型去调用，
但执行位置通常还是你这边；
真正由平台侧运行的，主要是 OpenAI 自带的 hosted tools。
